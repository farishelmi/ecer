Getting Started

To begin integrating your POS system with cloudECR, follow the steps outlined below. This section will guide you through the initial setup process, ensuring that you have all the necessary components and information to get started smoothly. By the end of this section, you will have a fully configured account and be ready to process transactions through cloudECR.

Prerequisites​

  • Merchant Application Account
  • Compatible POS systems
  • IP Whitelisting (Sandbox mode)
  • Internet connection

Merchant Application Account​

Merchants must register an application account with Fiuu before integrating with cloudECR. For integration and production processing, Fiuu will provide the following account information to the merchant:

ItemDescription
Secret KeyA server-side shared secret key used to generate signatures for API communication.
POS API AccountAn API login ID used to authenticate users to access protected APIs. Each POS is assigned a distinct ID.
Device IDA unique identifier for terminals. Each terminal is assigned a unique ID for identification purposes.
SECRET KEY

Each merchant is issued a unique Secret Key for signature validation.
You can obtain it in our Merchant Portal under cloudECR menu.
Store this key securely — it is required for authentication and other security processes.

Setting Up API Account​

  1. Log in to the Fiuu Merchant Portal (ensure you have completed merchant registration).
  2. Navigate to the cloudECR menu and click + New Account button.
  3. Check your merchant registered email for a verification link and follow it to activate the account.

    Note: If you don’t see the email, check your spam or junk folder.

Compatible POS Systems​

Our solution is designed to work with any POS system that can communicate via RESTful APIs.

  • Protocol: HTTP/HTTPS (REST API)
  • Data Format: JSON (UTF-8 encoded)
  • Authentication: Bearer token
  • Capabilities Required:
    • Ability to send POST requests with JSON payloads.
    • Ability to receive webhook/callback responses (optional for async flows).
    • Network access to cloudECR server’s API endpoint.

API Configuration:​

1. Service Environments​

  • Sandbox
    Used for testing and development purposes. All requests made to the sandbox URL will simulate real-world interactions without affecting the production system.

    URL: https:​//sandbox-cloudecr.fiuu.com

  • Production
    Used for real-world use. Requests to this URL will directly interact with the production system and should be handled with caution.

    URL: https:​//cloudecr.fiuu.com

2. Authentication​

Our API uses a two-token strategy for secure access:

  • Access Token – Obtained by logging in with your POS Account ID and Password.

    • Included in the Authorization header for each API request.
    • Expires: 15 minutes or at 12:00 AM (midnight), whichever comes first. This reduces risk if the token is exposed.

  • Refresh Token – Used to obtain a new Access Token without re-entering credentials.

    • Expires: at 12:00 AM (midnight).
    • When the Access Token expires, use the Refresh Token to request a new one.
    • If the Refresh Token has expired, you must log in again with your Merchant ID and Password.

Example login request:

POST /auth/login
{
  "posId": "your-pos-account-id",
  "password": "your-pos-account-password"
}

3. Request Format​

All API requests require a signed payload to ensure authenticity and integrity:

  • Signature – Generate using HMAC + SHA256 with your merchant Secret Key.
  • Content-Type – Always use application/json.
  • Required Header:
    • Authorization: Bearer access_token

Signature generation rules:

  1. Serialize the request body JSON values into a flat structure.
  2. Sort them in ascending order and concatenate the payload.
  3. Apply HMAC-SHA256 using Secret Key.
  4. Include the signature into request body.

See the Request section for full examples and signature generation code samples.

4. Response Format​

All API responses are returned in JSON format with a consistent structure to simplify parsing and error handling.

Standard response structure:

{
   "transType": "sale",
   "posId": "POS-ID",
   "deviceId": "Device-ID",
   "transData": {
      "transAmt": "10.00",
      ...
   },
   ...
}

Example error response:

{
  "code": "50301",
  "message": "Terminal not connected."
}

See the Response section for full examples.

5. Timeouts & Limits​

  • Request Timeout – Each API request will time out after 80 seconds if no response is received.

  • Rate Limits –

    • Sandbox: 30 requests per 15 minutes per POS account.
    • Production: 100 requests per 15 minutes per POS account.

  • Retry Recommendations –

    • Use exponential backoff for retries (e.g., 1s, 2s, 4s…).
    • Avoid retrying for 4xx errors (except 429 Too Many Requests).

  • Large Payloads – Maximum request body size is 1 MB.

6. Security Notes​

  • Always use HTTPS for all requests — unencrypted HTTP is not supported.
  • Keep your Secret Key, Access Token, and Refresh Token confidential — never store them in client-side code or public repositories.
  • Regenerate your Secret Key immediately if you suspect it has been exposed.
  • Sign all payloads using the HMAC-SHA256 method described in the Request Format section to prevent tampering.
  • Ensure all client systems maintain accurate timestamps — signature verification may fail if the request timestamp differs from server time beyond allowed tolerance.
  • Implement IP whitelisting where possible for an extra layer of protection.
  • Do not log full sensitive values (e.g., entire secret keys, full card numbers); mask or truncate them in logs.
